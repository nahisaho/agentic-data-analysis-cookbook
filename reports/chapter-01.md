# 第1章 なぜ今「AIエージェント時代のデータ分析」なのか

> **本章の到達目標**
> - 本書が「誰の」「どんな困りごと」を解決する本なのかを理解する
> - 「AIエージェント時代のデータ分析」が従来の分析とどう違うのかを説明できる
> - 本書の最終ゴール（合格ライン）と、それに至る道筋を把握する
>
> **この章で扱わないこと**
> - MCPやSkillの技術的な仕組み（第3章以降）
> - 環境構築やコード（第4章以降）

---

## 1.1 ある研究者の午後

半年前に作ったピーク解析のノートブックを開く。だが、どのセルをどの順番で実行すればよいか思い出せない。別の装置でとったデータに切り替えると、列名も単位もフォーマットも違い、また前処理を一から書き直す——。心当たりはないでしょうか。

実験データは日々増えていくのに、**分析の"やり方"は個人の頭とバラバラなノートブックの中にしか残っていない**。本書は、この状況を変えるための本です。

本書は、**ARIM（Advanced Research Infrastructure for Materials and Nanotechnology：マテリアル先端リサーチインフラ）データポータルの会員**、とりわけ次のような方に向けて書かれています。

- 実験装置からデータは出てくるが、**分析のたびに毎回スクリプトを書き直している**方
- Python や Jupyter を触った経験はあるが、**AIエージェントやMCPは使ったことがない**方
- 質量分析、分光、顕微鏡、回折・散乱など、**多様な装置データを横断的に扱いたい**方
- 「自分の実験データに合わせて分析を自動化したいが、どこから手をつければよいか分からない」方

本書のゴールは1つに絞っています。

> **読者が、自分の実験データに合わせて「データ分析用のSkill」を1つ以上、自力で作成できるようになること。**

これが本書の**合格ライン**です。単に既存の手順をなぞるのではなく、**あなた自身のデータ・装置・研究テーマに合わせて分析の仕組みを組み立て、繰り返し使える形で残せる**状態を目指します。

> [!NOTE]
> **Skill（スキル）とは何か**
> ここではまず、**「分析手順を再利用可能な形にまとめたもの」** と捉えてください。たとえば「スペクトルを読み込み、ベースライン補正し、ピークを検出して面積を比較する」という一連の手順を、入力・出力・制約・評価基準とともにパッケージ化したものです。AIエージェントとの具体的な関係は1.3および第3章で、作り方は第7章以降で扱います。

## 1.2 実験データ分析の「これまで」と「困りごと」

ARIMのデータポータルには、質量分析、クロマトグラフ、磁気共鳴、各種分光（状態分析）、表面分析、回折・散乱、光学顕微鏡、走査型顕微鏡、透過電子顕微鏡、成膜装置、リソグラフィ、各種特性評価など、**極めて多様な装置カテゴリのデータ**が蓄積されています[脚注1]。

しかし、実際のデータ分析の現場では次のような「困りごと」が繰り返されています。

| 困りごと | 具体例 |
|---|---|
| 分析の属人化 | 特定の研究者しか解析スクリプトを扱えない |
| 再現の難しさ | 半年前の解析手順を再現できない |
| 装置ごとの分断 | 装置ごとにフォーマットもツールもバラバラ |
| 前処理の繰り返し | 毎回同じ前処理コードをコピペしている |
| 知識の分断 | 解析結果と関連文献が結びついていない |

これらは「ツールが足りない」のではなく、**分析の"やり方"そのものを再利用可能な形で残せていない**ことに根本原因があります。本書はこの課題に、AIエージェントとSkillという2つの鍵で応えます。

## 1.3 「AIエージェント時代」に何が変わるのか

近年、大規模言語モデル（LLM）を中核とした**AIエージェント**が実用段階に入りつつあります[脚注2]。従来のプログラムが「決められた手順を実行するだけ」だったのに対し、AIエージェントは次の特徴を持ちます。

- **自然言語で指示できる**：「このスペクトルのピークを検出して、面積を比較して」と書くだけで分析が進む
- **道具（ツール）を自分で使う**：ファイル読み込み、可視化、計算などを状況に応じて選択・実行する
- **文脈を保ちながら反復する**：結果を見て次の手を考え、対話的に分析を深められる

そして、このAIエージェントが**外部の道具やデータにつながるための共通規格**が **MCP（Model Context Protocol）** です。

```text
【従来】
研究者 → 手書きスクリプト → 実行 → 結果を目視 → またスクリプト修正

【AIエージェント時代】
研究者 → 自然言語で指示 → AIエージェント ─(MCP)→ Jupyter / 文献 / 解析ツール
                              ↑____________ 結果を解釈して次の手を提案 ____________↓
```

> [!NOTE]
> **MCPとは何か**を一言でいうと、「AIエージェントと、外部ツール・データソースをつなぐための"共通コネクタ規格"」です[脚注3]。USB-Cが機器と周辺機器をつなぐ共通端子であるように、MCPはAIと道具をつなぐ共通端子だと考えてください。なお、MCPは"つなぐ"ための規格であり、**安全に使うための権限設計や承認ルールは別途必要**です（第6章で扱います）。

## 1.4 従来のデータ分析との違い

AIエージェントを使う分析は、従来の分析を**置き換える**ものではなく、**役割分担を変える**ものです。

| 観点 | 従来の分析 | AIエージェント時代の分析 |
|---|---|---|
| 指示の仕方 | プログラミング言語 | 自然言語 + 必要に応じてコード |
| 分析の主体 | 人間がすべて記述 | 人間が意図を示し、エージェントが実行 |
| 再利用単位 | スクリプト・関数 | **Skill（分析の型）**・MCPツール |
| 前処理 | 毎回書き直し | Skillとして定型化・再利用 |
| 文献との接続 | 手動で検索 | MCP経由で検索・取得を支援し、人間が妥当性を確認して統合 |
| 人間の役割 | 実装者 | **意図設計者・検証者** |

重要なのは、**人間の役割が「コードを書く人」から「意図を設計し、結果を検証する人」へ移る**という点です。AIエージェントは提案し、**最終判断は人間が行います**。この「Human-in-the-loop（人間が輪の中にいる）」の原則は本書を通じて一貫します。

## 1.5 装置カテゴリを「データ型」で捉える

ARIMの装置カテゴリは非常に多く、一つひとつを個別に扱うと学習が追いつきません。そこで本書では、装置カテゴリを**6つの「データ型」に抽象化**して考えます。同じデータ型なら、分析の基本パターンとSkillの骨格を共通化できるからです。

| データ型 | 代表装置カテゴリ |
|---|---|
| スペクトル型 | 各種分光（状態分析）、磁気共鳴、質量分析 |
| クロマトグラム・時系列型 | クロマトグラフ、熱分析、プロセス条件ログ |
| 画像・顕微鏡型 | 光学顕微鏡、走査型顕微鏡、透過電子顕微鏡 |
| 回折・散乱パターン型 | 回折・散乱、表面分析 |
| 表形式・プロセス条件型 | 成膜、リソグラフィ、機械/電気/磁気特性 |
| マルチモーダル統合型 | 上記の組み合わせ |

> [!NOTE]
> まず「自分が扱うデータはどの型か」を意識してください。この分類は第2章で詳しく行い、データ型ごとの適用テンプレートは第13章で提供します。1つの型でSkillが作れれば、同じ型の他装置にも**骨格を転用しやすくなります**。ただし、列名・単位・校正条件・メタデータは装置ごとに調整が必要です。

## 1.6 本書で使う標準環境

本書のハンズオンは、以下を標準環境とします（セットアップは第4章）。**最初からすべてを導入する必要はありません**。まず最小構成で動かし、必要に応じて発展構成を足していきます。

**最小構成（第II部で使う）**

- **Python + JupyterLab**：分析の実行基盤
- **GitHub Copilot CLI**：ターミナルで動くAIエージェント
- **Jupyter MCP**：ノートブックをAIエージェントから操作するためのMCPサーバ

**発展構成（第IV部以降で使う）**

- **ToolUniverse MCP**：科学ツール群をエージェントから呼び出すためのMCP
- **arXiv MCP / Paper Search MCP**：文献検索
- **FastMCP**：自作MCPの構築
- **scikit-learn系の解析ツール連携** など[脚注4]

これらを組み合わせることで、「**自然言語で実験データの基本的な分析を行える**」状態を作ります。各ツールの詳細な紹介と使い分けは付録Bのカタログにまとめます。

## 1.7 便利さとリスクはセットである

本書は便利さだけを説く本ではありません。AIエージェントによる分析には固有のリスクがあります。

> [!WARNING]
> - **循環設計問題**：AIに結果も評価基準も任せてしまい、正しさを検証できなくなる問題
> - **データ漏洩**：機密データを外部サービスへ送ってしまうリスク
> - **ハルシネーション**：もっともらしいが誤った解析・考察が生成されるリスク
> - **再現性欠如**：プロンプトや環境の差で結果が変わり、再現できなくなるリスク

本書はこれらを「予防」と「事例・検証」に分けて体系的に扱います。予防（安全ルール・Skill設計での回避）は第6・7章、実行後の検証と失敗事例は第12・14章です。本章の時点では「**便利さと注意はセット**」であることを覚えておいてください。

## 1.8 本書を読み終えると、あなたはこうなる

本書を最後まで実践すると、次の3段階を達成できます。

1. **自然言語で分析を進められる**（前処理・可視化・特徴抽出・考察）
2. **繰り返す分析をSkill化し、適切なMCPと組み合わせて再利用できる**
3. **検証・安全・再現性を担保して運用できる**——そして、自分のデータ向けに **Skillを1つ以上、自力で作成**できる（＝合格ライン）

各章の詳しい到達項目は、次の1.9の章構成表を参照してください。

## 1.9 本書の構成と読み進め方

本書は5部構成です。各部は「理解 → 最小実行 → Skill化 → 拡張・検証 → 運用」という学習曲線に沿っています。詳細な章の依存関係は各部の扉で示します。

| 部 | 章 | 内容 | この部を終えると |
|---|---|---|---|
| 第I部 なぜ必要か・何を作るか | 第1〜3章 | 目的・全体像・AI Agent/MCP/Skillの関係 | 何を作るのかが分かる |
| 第II部 最小環境で動かす | 第4〜6章 | 環境構築・最小分析・安全な使い方 | 自然言語分析を動かせる |
| 第III部 Skillを作る | 第7〜9章 | Skill設計・データ整形・単一データ解析Skill | 最初のSkillを作れる |
| 第IV部 Skillを拡張・検証する | 第10〜12章 | 文献統合・マルチモーダル・検証/レポート化 | Skillを実用レベルにできる |
| 第V部 横展開と運用 | 第13〜15章 | 装置別テンプレート・失敗事例・組織導入 | 自分の現場へ展開できる |
| 付録 | A・B・C | テンプレート集・MCPカタログ・トラブル対処 | 実務で参照できる |

### 前提知識と必要なもの

- **前提知識**：Pythonの基本文法とJupyterの基本操作（セルの実行、ライブラリのimport）。高度な機械学習の知識は不要です。
- **必要な環境**：ローカルPC（第4章でセットアップ）。インターネット接続、AIエージェント（GitHub Copilot CLI）の利用環境。

### 読み方の指針

> [!NOTE]
> - **最短で「動く」を体験したい**：第1・3・4・5章（ここまでで自然言語分析を体験）
> - **最初のSkillを作りたい**：第1〜9章を順に。特に**第6章（安全な使い方）と第8章（データ整形）は飛ばさない**でください。安全性とデータ契約は、実用的なSkillの土台になります。
> - **自分の現場へ展開したい**：第10章以降と第13章、付録A（テンプレート集）を参照

次章では、まず**ARIMデータと実験ワークフローの全体像**を俯瞰し、あなたのデータがどの「データ型」に当たるかを分類します。

## 本章のまとめ

- 本書はARIM会員が**自分のデータ向けにSkillを1つ以上自力で作れる**ようになることを目指す（合格ライン）
- 従来分析の課題は「分析のやり方を再利用可能に残せていない」ことにある
- AIエージェント時代は、**人間が意図を設計し、エージェントが実行、人間が検証する**役割分担へ移る
- 多様な装置カテゴリは**6つのデータ型**に抽象化して捉える
- 便利さとリスクはセットであり、注意事項は「予防」と「事例・検証」に分けて体系的に扱う

> **次章予告**：第2章では、ARIMの多様な装置データを横断する「共通の分析ワークフロー」を描き出し、あなたのデータのデータ型を分類します。
