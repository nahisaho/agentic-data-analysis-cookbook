# 第7章 データ分析用Skillの設計原則

> **本章の到達目標**
> - 良いSkillが備えるべき**6要素**（目的・入力条件・出力形式・成功条件・禁止事項・再現性条件）を挙げられる
> - 各要素を、自分の分析手順に対して**言葉で書き出せる**
> - **循環設計問題**（結果も評価もAI任せ）を、Skill設計の段階で回避する方法を説明できる
> - 章末の**Skill仕様書テンプレート**を、次章以降で自分のSkillを書き始めるための出発点として使える
>
> **この章で扱うこと／扱わないこと**
> - 扱う: Skill仕様の6要素、循環設計問題の回避、設計時セルフレビュー、Skill仕様書テンプレート
> - 扱わない: Skillの実際のファイル配置・コード（第9章のハンズオン）、入力データの契約詳細（第8章）、実行後の検証手法（第12章）、失敗事例と原因分析（第14章）、運用側の禁止操作・権限（第6章）

> [!IMPORTANT]
> 第3章で「Skill＝分析手順を再利用可能な形にまとめた型」と定義し、第5章で「まだSkillにしない理由」に触れました。本章はその2つをつなぎます。**動く分析を、再利用に耐える Skill に育てるための"設計図"** を書けるようにするのが本章のゴールです。実装は第9章以降で行います。

---

## 7.1 なぜ「設計」から始めるのか

第5章で自然言語だけの分析が動きました。この体験があると、「毎回同じ手順を書くのは面倒なので Skill にしよう」という気持ちが自然に湧きます。ここで**すぐ実装に走らない**のが本章の主題です。

設計をしないまま Skill を作ると、次のような Skill が量産されます。

- **入力条件があいまい** —— 「なんとなくスペクトルデータ」しか書かれておらず、単位・列名・欠損の扱いがバラバラ
- **出力形式が不定** —— 実行のたびに結果の形が変わり、後段で再利用できない
- **成功条件が曖昧** —— 「うまく動けばよい」で終わり、失敗を検出できない
- **禁止事項が抜けている** —— 生データを上書きする、機密データを外部送信する、といった事故が防げない
- **同じ入力でも結果が変わる** —— 乱数シード・パラメータの既定値が書かれておらず、再現性がない

こうしたSkillは、「動くけれど信じられない」道具になります。しかも一度公開してしまうと、**あなた自身も他の研究者も間違いに気づきにくくなります**。だから、コードを書く前に**言葉で設計**するのです。

> [!NOTE]
> Skill設計は、実験ノート冒頭の「目的・試料・条件・成功判定」に対応するものです。研究ノートを書く要領で、まず日本語で6要素を埋めるだけで、Skill の質が大きく変わります。

## 7.2 Skill仕様の6要素

第3章の Skill 定義（目的・入力・手順・出力・制約・評価基準）を、本章では**設計に必要な6要素**に整理し直します。「手順」は第9章のコードに任せ、代わりに**再現性条件**を独立させます。

```mermaid
flowchart TB
    subgraph Spec["Skill仕様書（6要素）"]
        P["① 目的<br/>何のためのSkillか"]
        I["② 入力条件<br/>受け付けるデータ・前提"]
        O["③ 出力形式<br/>何を返すか"]
        S["④ 成功条件<br/>何をもって成功と見なすか"]
        F["⑤ 禁止事項<br/>やってはいけないこと"]
        R["⑥ 再現性条件<br/>同じ入力で同じ結果が出るための決まりごと"]
    end
    P --> I --> O --> S --> F --> R
```

| # | 要素 | 一言でいうと | 例（スペクトル型ピーク検出Skill） |
|---|---|---|---|
| ① | **目的** | 何を解決するSkillか | 「1D スペクトルから主要ピークを検出し、位置・強度・幅を返す」 |
| ② | **入力条件** | どんなデータを受け付けるか | 「列 `wavenumber` / `intensity` を持つ CSV。単位 cm⁻¹、欠損なし」 |
| ③ | **出力形式** | 何をどう返すか | 「`peaks.csv`（列: `peak_id, center, intensity, prominence, fwhm`）」 |
| ④ | **成功条件** | どう判定すれば成功か | 「検出ピーク数が期待範囲内 かつ 既知標準試料で中心 ±2 cm⁻¹ 以内」 |
| ⑤ | **禁止事項** | 何をしてはならないか | 「生データ CSV の上書き、外部APIへの送信、単位変換の暗黙実施」 |
| ⑥ | **再現性条件** | 何を固定すれば同じ結果になるか | 「乱数シード=42、`find_peaks` の `prominence`/`distance` を明示、Skillバージョン、依存ライブラリ版」 |

> [!TIP]
> 6要素のうち、初学者が最も抜かしがちなのは **④成功条件** と **⑥再現性条件** です。この2つが埋まっているだけで、Skillの信頼度は大きく上がります。「動く」を超えて「信頼できる」に到達するための2要素だと覚えてください。

## 7.3 循環設計問題を設計で回避する

第1章と第3章で触れた**循環設計問題**を、ここで正面から扱います。

> **循環設計問題**：分析対象の結果を AI に出させ、その正しさの評価も AI に任せてしまい、**外部から検証できなくなる**問題。「AIの出力を、AIの出力で確かめている」状態。

Skill 設計の段階でこの問題を防ぐには、次の3つの分離を意識します。

```mermaid
flowchart LR
    A["分析（結果を出す）"]
    B["成功条件（判定基準）"]
    C["検証（判定を実施）"]
    Dev["開発用サンプル"]
    Eval["評価用サンプル<br/>（別に確保）"]
    A -.->|同じ Skill / 同じ AI にしない| B
    B -.->|Skillの外側で持つ| C
    Dev -.->|調整に使う| A
    Eval -.->|性能測定に使う| C
```

| 分離 | どう分けるか | 具体的なやり方 |
|---|---|---|
| **① 結果と判定基準を分ける** | 成功条件は Skill の外側（仕様書）に固定する | ④成功条件を**先に**書き、後から結果に合わせて書き換えない |
| **② 判定基準と検証者を分ける** | 判定は AI ではなく**人間・既知データ・別手段**が行う | 標準試料での中心位置比較、物理定数との一致、独立実装との突合 |
| **③ 学習データと評価データを分ける** | プロンプト調整で使った例と、性能を測る例は別にする | 開発用サンプル（synthetic-A）と評価用サンプル（synthetic-B, 実データ-C）を分けて残す |

> [!WARNING]
> 循環設計問題の本質は「AIに評価基準の下書きを出させたこと」自体ではなく、**結果を見た後に都合よく基準を採用・変更してしまう**こと、そして**人間が外部基準で基準を固定しない**ことです。AIを下書きに使うのは構いません。ただし、**最終的な成功条件は人間が外部基準（標準試料・物理定数・独立手段）で確定して仕様書に凍結し**、Skillはその条件を満たすかを事実で示す——この順番を崩さないでください。実行後にどう検証するかは第12章、実際に事故った例は第14章で扱います。

## 7.4 各要素の書き方

6要素を書くときの実務的なコツを、要素ごとにまとめます。

### ① 目的：1文で「何を」「いつ使うか」を書く

- 悪い例：「スペクトルを分析する」（何をどうするのか、いつ使うのか不明）
- 良い例：「Raman スペクトル（1D）から主要ピークを検出し、位置・強度・幅を表形式で返す。Ramanスペクトルのピーク位置を確認したいときに使う」

**主語（何を）・述語（どうする）・対象範囲（どこまで）** の3点に加え、**「いつ使うか（when to use）」** を必ず含めます。これは Agent Skills 仕様[脚注3]で `description` に「what it does **and** when to use it」の両方を含めることが求められているためです。実装時（第9章）にこの1文がそのまま `SKILL.md` の `description` フィールド（1024文字以内）になり、**Copilot はこの description を見て Skill を自動発見・起動判定します**[脚注4]。つまり「いつ使うか」の記述が甘いと、必要な場面で Skill が呼ばれません。

### ② 入力条件：受け付ける範囲と受け付けないものを両方書く

- 受け付ける：ファイル形式、列名、単位、値域、想定サイズ、必須メタデータ
- 受け付けない：欠損を含む、単位が違う、想定外の列がある、想定外の波数範囲

「**受け付けないものを検出したらエラー**」まで書けると、Skill が事故を起こしにくくなります。詳細な入力側のルール（データ契約）は第8章で扱います。

### ③ 出力形式：スキーマを固定する

- ファイル形式（CSV / Parquet / JSON など）、列名、単位、行順、必須メタデータ
- 図やプロットも出す場合は、ファイル名・形式（PNG/SVG）・軸の単位を仕様に含める
- 「表と図と要約テキストの3点セットを返す」のように**セット単位で決める**と、後段で使いやすい

### ④ 成功条件：客観的に検証できる形にする

- 数値で書く：「検出ピーク数が真値の ±1 本以内」「中心誤差が ±2 cm⁻¹ 以内」
- 参照を持つ：「既知標準試料（例: シリコン 520 cm⁻¹）で中心誤差 ±2 cm⁻¹」
- 数値化しにくい対象（画像判読・マルチモーダルの妥当性など）は、**事前に定義した人間レビュー項目**（チェックリスト）を使ってもよい
- ただし「AIが妥当と判断」「AIの説明が納得的」は成功条件になりません（循環設計問題）

### ⑤ 禁止事項：Skill仕様上の「やらない」を明文化する

- 第6章の**運用側の禁止**とは別に、Skill が**自らやってはならないこと**を書きます
- 例：「生データファイルの上書き」「単位変換を暗黙にやる」「入力にない列を勝手に補完する」「実行環境の外に結果を送信する」

### ⑥ 再現性条件：何を固定すれば同じ結果になるかを列挙

- 乱数シード、パラメータ既定値、依存ライブラリのバージョン、Skill 自身のバージョン
- CPU/GPU の切り替え、並列度・スレッド数（結果に影響する場合）
- 「同じと見なす**許容差**」を明示する（例：中心位置 ±0.01 cm⁻¹ 以内、面積 ±0.5% 以内）。浮動小数演算・BLAS・並列化の順序差により、**完全一致は保証できない**のが実情
- 「実行時に使ったバージョン・シードを結果ファイルにも記録する」まで書けると強い[脚注2]
- 環境の再現性は第4章の仮想環境と `pip freeze`、実行後の妥当性検証は第12章で扱います[脚注1]

### 仕様書と Agent Skills 仕様（SKILL.md）の対応

第9章で実装に進むとき、6要素は Agent Skills 仕様[脚注3]の `SKILL.md` に次のように対応します。設計段階でこの対応を意識すると、実装時の書き直しが減ります。

| 本章の要素 | SKILL.md での配置 | 補足 |
|---|---|---|
| **skill name** | frontmatter `name` | 1〜64文字・小文字英数字とハイフン・親ディレクトリ名と一致 |
| **① 目的**（what + when） | frontmatter `description` | 1024文字以内。「何を」と「いつ使うか」を必ず含める |
| **② 入力条件 / ③ 出力形式** | Markdown 本文（Examples of inputs/outputs） | 例で示すのが agentskills 推奨 |
| **④ 成功条件** | Markdown 本文（Common edge cases 等） | 数値・標準・チェックリストで具体化 |
| **⑤ 禁止事項** | Markdown 本文 + frontmatter `allowed-tools`（実験的） | 権限で担保する部分は第6章と連動 |
| **⑥ 再現性条件** | frontmatter `compatibility` + `metadata.version` + 本文 | 環境要件・シード・許容差 |
| **バージョン・作成日等** | frontmatter `metadata`（任意のキー） | 例: `metadata: { author: ..., version: "0.1" }` |

> [!TIP]
> `SKILL.md` 本文には長さの目安があります（推奨 500行以下 / 約 5000 トークン以下）。仕様が膨らんだら、詳細を `references/` サブディレクトリの別ファイルに切り出す前提で設計してください（**progressive disclosure**：メタデータ→本文→参照ファイルの順で必要になったときだけ読み込む方針）[脚注3]。

### Skillの配置場所と共有スコープ

Skillは配置ディレクトリによって**共有スコープ**が決まります。設計段階で「このSkillはチーム共有か、個人専用か」を先に決めておくと、実装時に迷いません[脚注4]。

| スコープ | 配置場所 | 用途 |
|---|---|---|
| **ワークスペース／プロジェクト**（チーム共有） | `.github/skills/<name>/`、`.claude/skills/<name>/`、`.agents/skills/<name>/` | リポジトリ経由でチームと共有。ソース管理下 |
| **個人**（ユーザー全体） | `~/.copilot/skills/<name>/`、`~/.claude/skills/<name>/`、`~/.agents/skills/<name>/` | 全プロジェクトから使える。リポジトリには入らない |

Copilot は agent モードで**この2箇所からSkillを自動発見し、`description` を見て起動タイミングを判定**します[脚注4]。したがって §7.4 ① で強調した「what + when to use」を description に書くことが**Skillが呼び出されるかどうか**を左右します。

> [!IMPORTANT]
> **未公開データや共同研究の秘匿情報を扱うSkillは、原則として個人スコープ**（`~/.copilot/skills/`）に置き、`.github/skills/` に置いてリポジトリと一緒に共有しないでください。Skill 本文に機密情報の断片（試料ID・組成・トークン等）を書いてしまうリスクを避けるためです。共有スコープの判断は第6章のデータ機密度ラベル運用と連動させてください。

## 7.6 設計時セルフレビュー：8つの問い

書き終えたら、次の8問に自分で答えてみます。1つでも「はい」と言い切れなければ、その要素を書き直します。

1. **目的**：この Skill は、他の Skill と役割が明確に違うか？（何でも屋になっていないか）
2. **入力条件**：想定外の入力が来たとき、Skill は**気付ける**か？
3. **出力形式**：出力を見ただけで、次の Skill・レポートで使えるか？
4. **成功条件**：**AIを介さずに**判定できる基準になっているか？
5. **禁止事項**：一番怖い事故（生データ破壊・機密送信など）は明示的に止まるか？
6. **再現性条件**：明日、別のPCで、同じ入力を渡したら（許容差の範囲で）同じ結果になるか？
7. **境界**：この Skill と、隣の Skill（例：データ読み込み、可視化、レポート化）の**責務は重複していないか**？
8. **Progressive disclosure**：本文が肥大化していないか？（実装時 `SKILL.md` 本文が 500行を超えそうなら、詳細を `references/` に分ける前提で設計する）[脚注3]

> [!TIP]
> 7の「境界」は、第9章以降でSkillを分割・連結するときに効いてきます。ひとつのSkillに責務を詰め込みすぎると、成功条件が書けなくなります。**「一つのSkillは一つの目的」**を目安に。

## 7.7 よくある落とし穴（設計時）

| 落とし穴 | なぜ起きるか | 予防 |
|---|---|---|
| **成功条件を後付けする** | 実装で得られた結果に合わせて基準を書いてしまう | ④は**先に書き**、実装で書き換えたくなったら理由をメモして仕様書に残す |
| **禁止事項を性善説で運用** | Skill仕様書に「気をつける」と書くだけで済ませる | Skill仕様書には**禁止の意図・対象・期待動作**を明記し、実際の権限設定は第6章で扱った運用側の仕組み（`--deny-tool` 等）で別途担保する。仕様書と設定はセット |
| **入力条件の"暗黙の前提"を書き忘れる** | 開発者本人には自明でも、他人には不明 | 単位・欠損有無・データ量・想定装置を必ず明記 |
| **1つのSkillに全部詰める** | 「便利にしたい」欲で読込〜レポートまで1本化 | データ契約（第8章）・分析（第9章）・レポート（第12章）で分ける |
| **AIに"総合評価"を書かせる** | LLMの説明が納得的に見えるため採用しがち | ④はあくまで数値・標準・独立手段で検証。AIの説明は補助資料 |

> [!IMPORTANT]
> これらの落とし穴に「はまってしまった実例」と「そこから何を学ぶか」は第14章にまとめます。本章は**はまらないための設計側の予防**に徹しました。

## 章末ワーク

1. 第2章で選んだ**自分のデータ型**に対して、7.5のテンプレートを埋めてみる（最低限、目的と入力条件・出力形式・成功条件の4つ）

4. 第5章で書いた `analysis.ipynb` の手順から、**①目的・②入力条件・③出力形式・④成功条件・⑤禁止事項・⑥再現性条件**の6要素を抽出してテンプレートに落とし込む（手順そのものは第9章で扱うので、ここでは書かない）

> ワーク4で書いた仕様書は、第9章「単一データ解析Skillを作る」の**出発点**になります。設計と実装を分けて進めるのが、良いSkillへの近道です。

## 本章のまとめ

- Skill は実装より先に**設計する**。設計は仕様書に**言葉で残す**
- Skill仕様は6要素：**①目的・②入力条件・③出力形式・④成功条件・⑤禁止事項・⑥再現性条件**
- **循環設計問題**は「結果と判定基準」「判定基準と検証者」「学習例と評価例」の3つの分離で予防する
- 成功条件は **AIを介さずに検証できる数値・標準、または事前定義した人間レビュー項目**で書く。禁止事項は Skill 仕様書に意図と期待動作を書き、実際の権限設定は第6章の運用側で担保する（仕様書と設定はセット）
- 7.5のテンプレートを埋め、7.6の8つの問い（Progressive disclosure を含む）でセルフレビューしてから、実装（第9章）に進む。仕様書の6要素は Agent Skills 仕様[脚注3]の `SKILL.md` に**そのまま対応する形**で設計しておく

> **次章予告**：第7章で書いた**入力条件**は、実際のデータでは単位・欠損・メタデータのばらつきに悩まされます。第8章では、それらを整えて「**Skillが受け取れる形（データ契約）**」に揃える作法を扱います。データを整えなければ、良い設計も動かせません。
